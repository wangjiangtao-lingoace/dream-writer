# P0 优化：模式统一 + 分支修复 + 上下文复用 + 批量生成

## 变因

Token 优化三阶段完成后，系统存在四个 P0 级问题：分支冲突 BUG、续写上下文膨胀、模式概念重叠、批量生成受限。本次一并修复。

## 改动

### 1. 修复 confirmPhase v2/imitation 分支冲突

`PipelineService.ts` 中 `pipelineVersion >= 2` 的 if 块和 imitation 分支是独立 if 块，当 v2 + imitation 时 `phase === "planning" && allConfirmed` 会同时命中两个分支。

- confirmPhase 第 195 行：追加 `&& config.mode !== "imitation"` 条件
- resumePipeline 第 313 行：追加 `&& config.mode !== "imitation"` 条件

### 2. ContinuationService 复用 ContextAssembler

轻量续写从全量上下文加载（~15,000 tokens）切换为 ContextAssembler 精简组装（~1,500 tokens/章），并引入 ChapterSummary 读写链路。

- ContinuationService.ts：删除 7 个冗余函数，使用 ContextAssembler 组装上下文
- writingPhase.ts：导出 `generateQuickSummary` 和 `generateChapterSummary`
- 续写完成后异步写入 ChapterSummary 表

### 3. 模式统一 standalone+continue → create

PipelineConfig.mode 改为 `"create" | "imitation"`，新增 `sourceType: "idea" | "content"` 字段。向后兼容旧 mode 值自动迁移。

- PipelineService.ts：PipelineConfig 接口变更 + startPipeline 迁移逻辑 + executePipeline 合并分支
- writingPhase.ts：`config.mode === "continue"` → `config.sourceType === "content"`
- NovelForm.tsx / DashboardPanel.tsx / PipelinePage.tsx：前端 config 传参更新

### 4. 批量生成突破 3 章限制

- writingPhase.ts：移除 `Math.min(..., 3)` 硬上限
- imitationPlans.ts：Zod 校验 `max(3)` → `max(999)`
- writingPhase.ts：每章完成后更新 PipelineJob.progress（章节级进度）
