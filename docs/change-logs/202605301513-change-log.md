# Pipeline 统一抽象层：消除双轨编排

## 变更原因

create 和 imitation 模式有两套完全独立的阶段编排逻辑，做着本质相同的事（生成资产 → 规划结构 → 写作），导致：
1. 新增阶段需要在两处同步修改
2. imitation 模式跳过了 consistency_check（质量校验缺失）
3. structuring 阶段是 imitation 独有，但 create 模式的 chapterOutlines 做了相同的事
4. 代码维护成本高，逻辑分散

统一后所有模式走同一条阶段链路：`analyze → assets → planning → chapterOutlines → consistency_check → writing`，差异通过配置参数控制。

## 改动

### 1. 统一 analyze 阶段

`server/src/services/pipeline/analyzePhase.ts`：合并 `executeAnalyzePhase` 和 `executeAnalyzePhase_continue` 为单一函数，内部根据 `config.sourceType` 分流：
- `"idea"`（默认）：使用 `novel.inspiration`
- `"content"`：加载已有章节内容（最近 10 章，最多 8000 字）
- `mode === "imitation"`：注入 `bookAnalysisContext` 作为参考
- 大纲生成：`sourceType === "content"` 使用 `generateOutlineFromChapters`，否则使用 `generateOutline`

### 2. 统一 planning 阶段

`server/src/services/pipeline/planningPhase.ts`：合并 `executePlanningPhase`（imitation）和 `executePlanningPhase_standalone`（create）为 `executePlanningPhase_unified`：
- 从 PhaseResult 加载前置阶段结果（outline、worldview、characters、style）
- 支持 `bookAnalysisContext` 和 `imitationPlanContext` 额外上下文注入
- 统一输出 `planning/volume_outline`
- 保留卷纲复用逻辑（DB 已有时直接加载）

### 3. 删除 structuring 阶段

`server/src/services/PipelineService.ts`：
- 删除 `executeStructuringPhase` 方法（约 50 行）
- imitation 模式不再走独立的 planning → structuring 路径
- 统一走 analyze → assets → planning → chapterOutlines → consistency_check → writing

### 4. 统一 confirmPhase 编排

`server/src/services/PipelineService.ts` confirmPhase：
- 删除 imitation 独立分支（planning → structuring → writing）
- 移除 `config.mode !== "imitation"` 条件，所有模式统一走 v2 分支
- consistency_check 对所有模式执行（imitation 之前跳过，现在补上）

### 5. 统一 executePipeline 入口

`server/src/services/PipelineService.ts` executePipeline：
- 合并 create/imitation 两个分支为单一入口
- 所有模式统一调用 `executeAnalyzePhase` 后暂停等待确认
- 统一 `totalSteps` 计算公式

### 6. 前端适配

`client/src/pages/PipelinePage.tsx`：
- `defaultPhases` 简化为按 `pipelineVersion` 判断，不再区分 imitation/create
- v2 统一使用 `["outline", "assets", "planning", "consistency_check", "writing"]`

## 修改文件

| 文件 | 改动 |
|------|------|
| `server/src/services/pipeline/analyzePhase.ts` | 合并两个分析函数为统一入口 |
| `server/src/services/pipeline/planningPhase.ts` | 合并两个规划函数为统一入口 |
| `server/src/services/PipelineService.ts` | 删除 structuring 阶段；统一 confirmPhase/executePipeline/resumePipeline |
| `client/src/pages/PipelinePage.tsx` | 简化 defaultPhases 映射 |

## 验证

- TypeScript 类型检查：server + client 均通过
- create 模式：完整 Pipeline 流程无变化
- imitation 模式：走统一路径（analyze → assets → planning → chapterOutlines → consistency_check → writing）
- 旧任务 resumePipeline：PhaseResult 命名兼容（structuring 映射保留）
- consistency_check 对 imitation 模式正常执行

## 风险说明

- **imitation 模式流程变化**：imitation 现在需要经过 analyze → assets → planning 三阶段确认，而非之前的一次性生成。这是有意为之的统一化改动。
- **历史任务兼容**：旧的 `structuring/*` PhaseResult 仍可通过 `regenerateStep` 中的映射逻辑重新生成。前端 translate.ts 保留了 structuring 的标签映射。
- **executeStructuringPhase 删除**：该方法已完全被 chapterOutlinesPhase + consistencyCheckPhase 替代，无遗留依赖。
