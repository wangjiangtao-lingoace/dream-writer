# 变更日志 - 节奏检测规则引擎

**日期**: 2026-07-03
**分支**: feat/lightweight-rebuild
**作者**: wangjiangtao

## 变更原因

小说创作流水线缺乏节奏层面的自动检测能力，无法在后处理阶段发现连续无爽点、情绪持续过高/过低、压抑无释放等节奏问题。新增节奏规则引擎，在章节后处理时自动运行 7 条节奏检测规则，将问题记录到日志。

## 修改点

### 新增文件

1. **`server/src/services/pipeline/pacingEngine.ts`** - 节奏检测规则引擎
   - 定义 `PacingRule` / `PacingContext` / `PacingIssue` 接口
   - 实现 7 条节奏规则：
     - `no_payoff_streak` - 连续无爽点（5 章 warning / 8 章 critical）
     - `payoff_too_dense` - 爽点过于密集（连续 2 章 warning）
     - `intensity_sustained_high` - 强度持续过高（连续 3 章 > 7 warning）
     - `intensity_sustained_low` - 强度持续过低（连续 3 章 < 3 warning）
     - `tension_no_release` - 压抑无释放（累积压抑 > 10 且爽点在 5 章前 critical）
     - `payoff_type_monotony` - 类型单一（最近 3 个爽点类型相同 warning）
     - `emotion_jarring_shift` - 节奏断裂（情绪差异 > 5 warning）
   - 导出 `checkPacing()` - 运行所有规则，返回 `PacingIssue[]`
   - 导出 `buildPacingSuggestion()` - 生成节奏建议文本，用于 Prompt 注入

### 修改文件

2. **`server/src/services/pipeline/postProcessing.ts`** - 后处理流程集成
   - 新增 `import { checkPacing } from "./pacingEngine"`
   - 在 `mergedPostProcessing` 尾部新增节奏检测调用（不阻塞主流程）
   - critical 级别问题输出 `console.warn`，所有问题输出 `console.info`

## 风险说明

- **低风险**：节奏检测以非阻塞方式运行（`.then().catch()`），失败不影响后处理主流程
- **数据依赖**：规则检查依赖 `PleasurePoint` 和 `EmotionCurve` 表数据，若数据为空则部分规则自动跳过
- **无副作用**：引擎仅读取数据库，不写入任何数据
