# 因果链管理功能

**变更时间**: 2026-07-03 00:51
**变更类型**: 功能新增
**影响范围**: Pipeline 后处理、上下文组装、数据库 Schema

## 变更原因

为 Dream Writer 系统新增因果链管理功能，用于追踪剧情事件之间的因果关系，帮助 AI 在续写时理解剧情脉络，保持逻辑连贯性。因果链是继伏笔、钩子、爽点链之后的又一剧情追踪维度。

## 修改点

### 1. 数据库 Schema 新增 CausalLink 模型

在 `server/prisma/schema.prisma` 中新增 `CausalLink` 模型（表名 `causal_links`），包含以下字段：

- `causeType` / `causeRefId` / `causeDescription`: 原因侧信息（事件、钩子、伏笔、冲突等）
- `effectType` / `effectRefId` / `effectDescription`: 结果侧信息
- `strength`: 因果关联强度（1-10）
- `isMainConflict`: 是否属于核心矛盾因果链
- `novelId` / `chapterOrder`: 索引字段，支持按作品和章节快速查询

### 2. 新增因果链服务

创建 `server/src/services/pipeline/causalChainService.ts`，实现三个核心函数：

- `captureCausalLinks(novelId, chapterOrder, result)`: 从后处理结果中自动提取因果关系并写入 DB。提取策略包括：相邻事件配对、钩子/伏笔回收追踪、角色成长关联、核心矛盾推进。
- `buildCausalChainContext(novelId, chapterOrder)`: 构建因果链上下文文本（约 200-400 tokens），注入续写 Prompt，帮助 AI 理解剧情脉络。
- `checkCausalChainIntegrity(novelId, chapterOrder)`: 检查因果链完整性，检测孤立事件、未解决伏笔（超过 10 章未回收）、悬挂因果等问题。

### 3. Pipeline 集成

- **后处理** (`postProcessing.ts`): 在 `mergedPostProcessing` 函数中，章节后处理完成后调用 `captureCausalLinks` 提取因果关系（不阻塞主流程）。
- **上下文组装** (`contextAssembler.ts`): 在 `assembleForChapter` 方法中，新增第 18 个维度——因果链上下文，与其他上下文并行加载。

## 风险说明

1. **数据库写入量增加**: 每章后处理会新增若干 CausalLink 记录，但数据量可控（每章约 3-8 条），不影响性能。
2. **上下文 Token 增加**: 因果链上下文约 200-400 tokens，在可接受范围内。
3. **依赖后处理结果**: `captureCausalLinks` 依赖 `parsed.chapterSummary.keyEvents` 等字段，如果后处理 LLM 返回异常，因果链提取可能为空，但不影响主流程。
