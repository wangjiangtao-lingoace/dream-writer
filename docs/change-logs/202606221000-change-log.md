# 202606221000 - P0 Token 预算上限 + 自动续写批次限制

## 变更原因

Pipeline 写作阶段存在两个 P0 级业务流缺陷：
1. Token 消耗无硬上限，用户可启动大规模 Pipeline（300 章、1.2-2.4M tokens）导致 runaway costs
2. `autoContinue` 模式下无用户可控的停止点，持续生成无背压机制

## 修改点

### P0 #1: Token Budget Hard Cap
- 前端 PipelineConfigModal 新增 "Token 预算上限" 输入框（单位 1K tokens），预估超限时显示警告
- 后端 PipelineConfig 新增 `tokenBudget` 字段（单位 1K tokens）
- writingPhase 在每章生成前检查累计 token 估算（中文字符 * 1.5 + 1500 prompt 开销），超限则暂停 Pipeline 并设置 `currentStep: "token_budget_reached"`

### P0 #2: Auto-Continue Backpressure
- 前端 PipelineConfigModal 新增 "每批次最多写入章数" 输入框（0 = 不限制）
- 后端 PipelineConfig 新增 `maxChaptersPerBatch` 字段
- writingPhase 在 `executeWritingPhase` 中限制 `draftCount`，批次完成后若还有剩余章节则暂停并提示 "已完成本批次 N 章"
- autoScheduler 的 `scheduleNextBatch` 也尊重 `maxChaptersPerBatch` 限制

## 文件列表

| 文件 | 变更说明 |
|------|----------|
| `client/src/components/PipelineConfigModal.tsx` | 新增 `tokenBudget` / `maxChaptersPerBatch` 字段、状态、UI 输入框、超限警告、handleConfirm 传递 |
| `server/src/services/PipelineService.ts` | PipelineConfig 接口新增 `tokenBudget` / `maxChaptersPerBatch` 可选字段 |
| `server/src/services/pipeline/writingPhase.ts` | token 预算累计追踪 + 章前检查 + 批次限制 + 暂停状态设置 + 返回 `{ chapters, budgetReached }` |
| `server/src/services/pipeline/autoScheduler.ts` | `scheduleNextBatch` 限制 batchSize 不超过 `maxChaptersPerBatch` |

## 风险说明

- Token 估算基于 `chars * 1.5 + 1500`，精度约 +/-30%，目标是防 runaway costs 而非精确记账
- `executeWritingPhase` 返回类型从 `void` 变为 `{ chapters, budgetReached }`，但所有调用方均为 fire-and-forget，无破坏性
- 暂停后恢复（resumePipeline）会从上次中断位置继续，token 累计重新开始计算（保守策略）
