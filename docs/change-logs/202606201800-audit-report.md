# Dream Writer 业务流程审查报告 20260620

> 审查日期：2026-06-20
> 审查视角：产品经理 / 业务架构师 / 最终用户
> 审查范围：业务流程、异常流程、运营流程、数据流转
> 审查方法：代码走读 + 架构分析（不审查代码风格，仅审查业务逻辑）

---

## P0 — 必须修复（影响核心功能可用性）

### 1. Pipeline 服务器重启后永久卡死
**视角**: 运营 / 业务架构师
**问题**: PipelineJob 以 `status="running"` 持久化到 SQLite，但服务器重启后没有任何恢复机制。所有进行中的 Pipeline 永久停留在 `running` 状态，无法继续、无法重试、用户只能看到"进行中"但永远无法完成。
**影响**: 生产环境中任何一次服务器重启/部署都会导致所有正在运行的创作任务丢失。
**建议**: 启动时扫描 `running` 状态的 PipelineJob，标记为 `error` 并记录原因，允许用户重试。

### 2. Pipeline 并发覆盖用户编辑
**视角**: 最终用户
**问题**: Pipeline 写作阶段（WritingPhase）直接 `upsert` 章节内容，不检查用户是否正在编辑。用户在工作台手动修改章节后，Pipeline 可能会覆盖其编辑内容，且无任何提示。
**影响**: 用户辛苦修改的内容被静默覆盖，且无法恢复。
**建议**: Pipeline 写入前检查章节最后修改时间，若用户在 Pipeline 启动后修改过，跳过或提示确认。

### 3. Pipeline 失败无回滚
**视角**: 最终用户
**问题**: Pipeline 任意阶段失败后，已完成阶段产生的数据（卷、大纲、角色、世界观）保留在数据库中，但 Pipeline 状态标记为 `error`。用户无法从失败点恢复，只能重新运行整个 Pipeline，导致数据重复或不一致。
**影响**: 一个 10 卷 300 章的 Pipeline 在第 4 阶段失败，前 3 阶段的数据变成"孤儿"，用户无法利用。
**建议**: 实现阶段级 checkpoint，支持从失败阶段重试而非全流程重跑。

### 4. 没有成本控制机制
**视角**: 运营 / 产品经理
**问题**: Pipeline 每个章节写作消耗约 4000-8000 tokens（含 context），一个"史诗"规模（10卷×30章=300章）的完整 Pipeline 消耗约 120-240 万 tokens。没有任何速率限制、预算上限、或消耗预估。
**影响**: 用户可能无意中触发巨额 API 消耗，且无法提前知晓成本。
**建议**: 在 PipelineConfigModal 中展示预估 token 消耗；Pipeline 执行中实时累计并展示已消耗 tokens；支持设置每日/单次预算上限。

### 5. 章节内容丢失风险（写入冲突）
**视角**: 最终用户
**问题**: `WritingPhase` 通过 SSE 流式写入章节内容，但 `handleChapterStreaming` 中如果 `targetIndex` 为 -1（章节未找到），会直接 `reject`。此时 Pipeline 中止，但已生成的部分内容未保存。
**影响**: 30 章的写作如果第 28 章因为索引问题失败，前 27 章的 ChapterSummary 可能未正确生成，后续无法续写。
**建议**: 写入失败时保存已生成的部分内容到草稿，而非直接丢弃。

## P1 — 应该修复（影响用户体验和系统健壮性）

### 6. 一致性校验发现问题但无法自动修复
**视角**: 最终用户
**问题**: `ConsistencyPhase` 能检测出时间矛盾、人物性格不一致、伏笔未回收等问题，但只生成 `ConsistencyIssue` 记录，不自动修复，也不阻断后续写作。用户需要手动在 ConsistencyPanel 中逐一处理，但没有"一键修复"或"AI 建议修复"功能。
**影响**: 一致性问题随章节增加而累积，最终可能导致严重的故事逻辑断裂。
**建议**: 对可修复的 issue（如时间线矛盾）提供 AI 辅助修复建议；对高严重度 issue 在继续写作前弹窗提示。

### 7. Hook/Foreshadow 生命周期断裂
**视角**: 产品经理 / 最终用户
**问题**: Pipeline 的"钩子与伏笔规划"阶段会创建 Hook 和 Foreshadow 记录（status="planted"），但后续写作阶段和一致性校验阶段都不会更新其状态为 "resolved" 或 "paid_off"。系统中存在大量状态为 "planted" 但实际上已经在章节中解决的伏笔。
**影响**: "未回收伏笔"列表永远不准确，用户无法依赖它来追踪故事线索。
**建议**: 在一致性校验阶段增加伏笔回收检测（AI 分析章节内容是否包含伏笔回收），自动更新状态。

### 8. 暂停不是即时的
**视角**: 最终用户
**问题**: 用户点击"暂停"后，Pipeline 不会立即停止——当前章节的 SSE 流会继续到完成。如果目标字数是 2500 字但 AI 已经写了 1200 字，暂停命令要等当前章节写完（可能还需要多次 continue）才生效。
**影响**: 用户无法在发现问题时快速中断。
**建议**: 前端暂停按钮点击后立即断开 SSE 连接，后端检测到连接断开后停止当前阶段。

### 9. Pipeline 进度展示不精确
**视角**: 最终用户
**问题**: Pipeline 前 4 个阶段（Outline/Assets/Planning/Consistency）是同步执行，用户看到的进度条在这些阶段会"卡住"很久，因为后端在同步处理但前端没有阶段级进度反馈。只有 WritingPhase 有 SSE 实时进度。
**影响**: 用户可能以为系统卡死而刷新页面，导致 Pipeline 状态异常。
**建议**: 前 4 个阶段也通过 SSE 推送阶段级进度（如"正在生成角色…3/8"）。

### 10. RAG 索引不完整
**视角**: 业务架构师
**问题**: `RagRetrieveService` 只索引了章节内容（通过 `upsertChapterChunks`），但没有索引世界观文档、角色档案、知识资产。Pipeline 的 WritingPhase 的 context 中虽然从多个子系统收集信息，但 RAG 检索只能匹配章节内容。
**影响**: 用户问"主角的身世是什么"时，RAG 无法从世界观文档中找到答案，只能依赖硬编码的 `knowledgeAssetBlock`。
**建议**: 将世界观、角色、知识资产也纳入 RAG 索引。

### 11. 内存系统（Memory）未被 Pipeline 使用
**视角**: 产品经理
**问题**: Memory 表有 4 层（permanent/long-term/short-term/temporary）和重要性衰减机制，但 Pipeline 的 ContextAssembler 中完全没有引用 Memory。Memory 只在用户手动交互时使用。
**影响**: 花费复杂度实现的记忆系统在自动化创作流程中完全闲置。
**建议**: Pipeline 写作阶段应将相关 Memory 纳入 context。

### 12. 章节续写没有上下文感知
**视角**: 最终用户
**问题**: `handleContinueWriting` 使用 `model.generateText(prompt)` 直接调用 LLM，不经过 ContextAssembler。续写不包含前文摘要、角色信息、伏笔状态等结构化上下文。
**影响**: 续写的内容可能与前文脱节，角色性格不一致。
**建议**: 续写应复用 Pipeline 的 ContextAssembler 逻辑。

## P2 — 可以优化（影响系统可维护性和扩展性）

### 13. 孤儿数据无清理机制
**视角**: 业务架构师
**问题**: 以下数据在 Pipeline 失败或重跑时可能变成孤儿：
- `ChapterOutline`（卷删除时级联删除，但 Pipeline 重跑时旧大纲不会清理）
- `ChapterSummary`（章节删除时未级联删除）
- `RagChunk`（章节删除时未级联删除）
- `KnowledgeAsset`（删除 Novel 时未级联删除）
- `Memory`、`StyleProfile`、`StoryState`（删除 Novel 时未级联删除）

**影响**: 长期运行后数据库膨胀，RAG 检索结果包含已删除内容的碎片。
**建议**: 补全 Prisma schema 中的 `onDelete: Cascade` 关系。

### 14. 风格分析（StyleProfile）只在创建时运行一次
**视角**: 产品经理
**问题**: 仿写模式下，`AnalysisCreate` 页面会分析源文并生成 StyleProfile，但这个分析只运行一次。用户后续修改了源文或想调整风格，无法重新分析。
**影响**: 风格锁定在初始分析结果，无法迭代优化。
**建议**: 提供"重新分析"按钮。

### 15. 健康检查端点过于简单
**视角**: 运营
**问题**: `/api/health` 只检查数据库连接，不检查 LLM API 可用性、磁盘空间、活跃 Pipeline 数量等关键指标。
**影响**: 运维无法通过健康检查发现 LLM 服务不可用等常见问题。
**建议**: 增加 LLM ping、磁盘空间、活跃任务数等检查项。

### 16. Pipeline 阶段配置不够灵活
**视角**: 产品经理
**问题**: 5 个阶段的顺序和启停是硬编码的，用户无法跳过某个阶段（如跳过一致性校验直接进入写作），也无法调整阶段参数（如校验严格度）。
**影响**: 高级用户无法根据需求定制 Pipeline 行为。
**建议**: 在 PipelineConfigModal 中增加阶段级开关和参数配置。

### 17. 缺少创作数据导出功能
**视角**: 最终用户
**问题**: 用户无法导出完整的小说数据（含章节、大纲、角色、世界观、伏笔等）。只能复制粘贴纯文本。
**影响**: 用户无法备份创作成果或迁移到其他平台。
**建议**: 提供 JSON/Markdown/DOCX 格式导出。

### 18. BookAnalysis 章节解析依赖 AI 而非结构化提取
**视角**: 业务架构师
**问题**: 用户上传 TXT 文件后，`BookAnalysisService.analyzeChapters` 用 AI 将全文按 `第X章` 格式拆分。如果原文不使用标准章节标记（如用"Chapter X"或数字标题），解析会失败或不准确。
**影响**: 非标准格式的书籍无法正确拆分章节。
**建议**: 在 AI 拆分前增加正则预扫描，支持多种章节标记格式。

### 19. 前端状态与后端不同步
**视角**: 最终用户
**问题**: NovelWorkspace 中的 `activeTab`、`activeVolumeId`、`activeChapterId` 等状态仅存在前端内存中，刷新页面后丢失。没有 URL 路由参数持久化这些状态。
**影响**: 用户刷新页面后回到默认 Tab，需要重新找到之前的章节。
**建议**: 将工作台状态同步到 URL 查询参数。

### 20. 错误信息不够用户友好
**视角**: 最终用户
**问题**: Pipeline 失败时 `phaseResults` 中存储的是原始错误堆栈，前端 PipelinePhaseCard 直接展示 `lastResult.error`。用户看到的是 `"Cannot read properties of undefined (reading 'content')"` 这样的技术错误。
**影响**: 用户无法理解失败原因，也不知道如何处理。
**建议**: 后端对常见错误进行分类，前端展示用户友好的错误描述和建议操作。

## 总结

| 等级 | 数量 | 关键主题 |
|------|------|----------|
| **P0** | 5 | 重启恢复、并发覆盖、失败回滚、成本控制、内容丢失 |
| **P1** | 7 | 一致性修复、伏笔生命周期、暂停延迟、进度展示、RAG 不完整、Memory 未使用、续写无上下文 |
| **P2** | 8 | 孤儿数据、风格重分析、健康检查、阶段灵活度、数据导出、章节解析、状态持久化、错误友好 |

**最高优先建议**: P0 #1（重启恢复）和 P0 #2（并发覆盖）应立即修复，这两个问题在生产环境中会导致数据丢失。P0 #4（成本控制）应在正式对外发布前完成。
