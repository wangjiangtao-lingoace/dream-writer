# P0/P1 优化：自动推进 + 配置弹窗 + 入口集成

## 变更原因

产品评审发现三个高优先级问题：
1. **P0: 确认流程过重** — 5 卷小说需要 9 次手动确认才能开始写作，用户被"审批员"化
2. **P0: imitation 硬编码 1 卷 3 章** — 拆书仿写只能生成 3 章，无配置入口
3. **P1: 创建表单缺少配置** — volumeCount/chaptersPerVol/targetWordCount 硬编码，用户无法控制创作规模

## 改动

### 1. autoContinue 自动推进机制

`autoContinue` 字段已存在于 PipelineConfig 但从未被读取（死字段）。现在所有阶段函数在生成完毕后检查该字段：
- `autoContinue=true`：自动确认当前阶段结果，直接进入下一阶段，不暂停
- `autoContinue=false`（默认）：行为不变，暂停等待用户确认

实现方式：`pipelineUtils.ts` 新增 `autoAdvanceOrPause(jobId, phase, nextPhaseFn)` 共享函数，各阶段函数在暂停点调用它替代原来的硬暂停。

### 2. PipelineConfigModal 配置弹窗

新建 `client/src/components/PipelineConfigModal.tsx`，提供统一的 Pipeline 启动配置界面：
- **自动推进开关**：默认开启，AI 自动完成所有阶段
- **创作规模预设**：短篇/中篇/长篇/史诗/自定义（create 模式）；短篇/中篇/长篇/自定义（imitation 模式）
- **自定义参数**：卷数(1-10)、每卷章数(5-30)、目标字数/章(500-8000)
- **自动草稿章数**：1-10，默认 3
- **覆盖已有章节开关**：默认关闭

### 3. 所有 Pipeline 启动入口集成配置弹窗

所有启动 Pipeline 的前端入口不再硬编码配置参数，改为弹出 PipelineConfigModal：
- `NovelForm.tsx`："开始创作"按钮 → 弹窗 → 创建作品 + 启动 Pipeline
- `DashboardPanel.tsx`：standalone/continue/draft 三个动作 → 弹窗 → 执行
- `AnalysisPanel.tsx`：applyPlanToPipeline → 弹窗 → 执行

## 验证

- TypeScript 类型检查：server + client 均通过
- autoContinue=true：Pipeline 从 analyze 一口气跑到 writing，中间不暂停
- autoContinue=false：行为与之前一致，逐步暂停确认
- 配置弹窗：所有 Pipeline 启动入口都弹出弹窗，用户可修改参数
- imitation 模式：不再硬编码 1 卷 3 章，弹窗提供短篇/中篇/长篇预设 + 自定义

## 风险说明

- **autoContinue 异常恢复**：如果自动推进过程中某阶段报错，Pipeline 进入 error 状态，用户可通过 resumePipeline 恢复。自动推进不会跳过错误处理。
- **弹窗体验**：用户每次启动 Pipeline 都需要经过弹窗，增加了 1 步操作。但这是有意为之——让用户明确知道自己在启动什么规模的创作。
- **autoConfirmByUser: false**：自动推进的确认标记为 `confirmedByUser: false`，与用户手动确认区分，便于后续分析。
