# 百万字长篇可持续性修复：RAG 上下文 + 状态追踪 + 章纲修正 + 字数修复

## 变更原因

产品评审发现系统无法支撑百万字长篇自主创作，核心缺陷：
1. 前文回顾只有 5 章窗口，第 300 章写时看不到第 1 章的设定
2. 角色/钩子/伏笔状态不追踪，写完即忘
3. 章纲预生成后不根据实际写作调整
4. 每章字数默认 1800，低于 2000-3000 的目标

## 改动

### 1. ContextAssembler 接入 RAG 检索 + Memory 表 + StoryState

`server/src/services/pipeline/contextAssembler.ts`：
- 新增 `loadRagContext()`：用章纲关键词调用已有的 RagRetrieveService，检索任意历史章节的相关上下文（之前完全未接入）
- 新增 `loadRelevantMemories()`：从 Memory 表加载重要度 >= 6 的记忆（角色关系、关键剧情、世界观规则等）
- 新增 `loadStoryState()`：加载当前剧情状态（主角处境、目标、剧情阶段、活跃伏笔）
- `buildCompactContext` 新增三个区块：【历史参考】【重要记忆】【剧情状态】

效果：第 500 章写作时能检索到第 1 章的关键设定，前文回顾从 5 章扩展到"按需检索任意历史章节"。

### 2. 每章写作后自动更新角色/钩子/伏笔状态

`server/src/services/pipeline/postProcessing.ts`：
- 新增 `updateCharacterAndTrackerStatus()`：分析章节内容，自动更新 Character.arcSummary、Hook.status、Foreshadow.status

`server/src/services/pipeline/writingPhase.ts`：
- 每章写完后调用 `updateCharacterAndTrackerStatus`
- 后处理（StoryState + Memory 提取）从仅续写模式扩展到所有模式

### 3. 章纲修正机制

`server/src/services/pipeline/writingPhase.ts`：
- 新增 `correctNextVolumeOutlines()`：每卷写完后检查实际内容与计划章纲的偏离度
- 偏离度 >= 5 时，用 LLM 修正下一卷的章纲，确保承接上一卷的实际走向
- 修正后的章纲直接更新 PhaseResult，后续写作使用修正版

### 4. 每章字数 2000-3000 字

- `writingPhase.ts`：默认 targetWordCount 从 1800 → 2500，maxTokens 从 2000-4500 → 3000-5000
- `ContinuationService.ts`：默认 targetWordCount 从 1800 → 2500，同步调整
- `PipelineConfigModal.tsx`：所有预设的 targetWordCount 从 1500-2000 → 2000-2500
- 写作 prompt 增加字数范围约束（不少于 80%，不超过 120%）

## 验证

- TypeScript 类型检查：server + client 均通过
- RAG 检索：需 ENABLE_RAG=true 环境变量启用，未启用时优雅降级（返回空字符串）
- Memory 加载：无记忆时返回空数组，不影响上下文
- StoryState：无状态时返回 null，不影响上下文
- 章纲修正：偏离度 < 5 时跳过，不影响正常流程

## 风险说明

- **RAG 依赖**：需 ENABLE_RAG=true + 嵌入服务可用。未启用时降级为原有行为（仅 5 章回顾）
- **章纲修正的 LLM 调用**：每卷多一次 LLM 调用（偏离度检查 + 可能的修正），增加约 1000 tokens 成本
- **Character.arcSummary 更新**：由 LLM 自动更新，可能与人工编辑冲突。建议后续增加"人工锁定"机制
