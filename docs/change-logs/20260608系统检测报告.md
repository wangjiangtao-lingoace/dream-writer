## 第一部分：产品定位分析

### 核心用户画像

| 维度 | 分析 |
|------|------|
| **核心用户** | 网文作者（起点/番茄/七猫），日更3000-10000字，长篇百万字级别 |
| **次要用户** | 短篇/中篇创作者、剧本杀编剧、同人作者 |
| **用户痛点** | ① 卡文（不知道写什么）② 设定前后矛盾 ③ 大纲到正文的鸿沟 ④ 每日更新压力 |

### 当前功能 vs 用户需求匹配度

| 功能 | 匹配度 | 问题 |
|------|--------|------|
| Pipeline 自动创作 | ★★★★☆ | 核心卖点清晰，但用户缺少中间干预点 |
| 世界观/人物管理 | ★★★☆☆ | 功能存在但数据提取不够丰满（已修复prompt限制） |
| 拆书模仿 | ★★★★★ | 独特差异化功能，网文作者刚需 |
| 记忆系统 | ★★★★☆ | 架构优秀，四层衰减模型合理 |
| 知识库/RAG | ★★☆☆☆ | MiMo不支持embedding，当前实际不可用 |
| 一致性校验 | ★★★☆☆ | 框架存在但校验规则不够丰富 |

### 优势
- **完整创作闭环**：从灵感到成书的全流程，竞品中罕见
- **拆书模仿能力**：分析参考作品风格并模仿创作，网文作者核心需求
- **智能记忆系统**：四层记忆+自动衰减，解决长篇创作的一致性问题
- **BYOK模式**：用户自带API Key，降低运营成本

### 问题
- **功能堆砌严重**：29个数据库模型、31个路由、45个服务文件——MVP阶段过度设计
- **伪需求**：`DirectorService`（AI导演）、`AutomationService`（自动调度）在用户还没学会用基础功能前无意义
- **MVP不清晰**：用户第一次打开看到14个tab，认知负荷过重

### MVP 应该是什么

```
核心MVP = 创建作品 → 输入灵感 → 一键生成大纲+人物+世界观 → 逐章写作 → 导出
```

砍掉的功能（移到v2+）：知识库、记忆系统、一致性校验、拆书、自动化调度、雷达图、情绪曲线

## 第三部分：AI Agent 架构分析

### 当前架构：非Agent化

当前是**硬编码Pipeline**，不是Agent架构。`PipelineService`按固定顺序执行5个phase，每个phase内的步骤也是硬编码的。

### 是否适合Agent化？

**适合**，因为：
- 小说创作是开放性任务，需要动态决策
- 不同作品需要不同的创作策略
- 用户干预需要Agent具备暂停/恢复/转向能力

### 推荐Agent架构

```
┌──────────────────────────────────────────────────────┐
│                   总控Agent (Orchestrator)             │
│  职责：理解用户意图，选择执行策略，协调子Agent          │
│  输入：用户指令 + 作品状态 + 历史上下文                 │
│  输出：执行计划 + 进度报告                              │
├────────────┬────────────┬────────────┬───────────────┤
│ 大纲Agent  │ 世界观Agent │ 人物Agent  │ 写作Agent     │
│ 结构规划    │ 设定构建     │ 角色塑造    │ 正文生成      │
├────────────┴────────────┴────────────┴───────────────┤
│                   校验Agent (Critic)                   │
│  职责：审查质量、检测矛盾、提出修改建议                  │
├──────────────────────────────────────────────────────┤
│                   记忆Agent (Memory)                   │
│  职责：维护上下文一致性，管理知识检索                    │
└──────────────────────────────────────────────────────┘
```

### 当前代码与Agent架构的差距

| Agent能力 | 当前状态 | 差距 |
|-----------|---------|------|
| 任务理解 | ❌ 无 | 只有硬编码的phase序列 |
| 动态规划 | ❌ 无 | 不能根据中间结果调整策略 |
| 工具调用 | ⚠️ 部分 | LLM调用有，但没有标准化的工具接口 |
| 记忆 | ✅ 有 | 四层记忆系统设计良好 |
| 自我反思 | ⚠️ 基础 | 质量校验是程序化的，不是LLM驱动的 |
| 多Agent协作 | ❌ 无 | 所有逻辑在单一Pipeline中 |

## 第四部分：任务编排分析

### 当前流程

```
灵感 → analyze(分析) → decompose(拆解) → outline(大纲)
  → assets(世界观+人物+风格) → planning(卷纲+章纲)
  → consistency(一致性检查) → writing(逐章写作)
```

### 问题分析

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 步骤过多 | ★★★☆☆ | 用户输入灵感后要等5+个AI步骤才能看到正文 |
| 缺少快速预览 | ★★★★☆ | 用户想先看"第一章长什么样"，但要等全部资产生成完 |
| 重复生成 | ★★☆☆☆ | analyze和assets阶段对世界观/人物有重复提取 |
| 无中间干预 | ★★★★☆ | 用户不能在"世界观生成后"手动修改再继续 |

### 推荐最佳工作流

```
Phase 0: 快速预览（30秒）
  灵感 → 生成3个开头方案供用户选择

Phase 1: 核心资产（2分钟，可并行）
  灵感 → [大纲Agent | 世界观Agent | 人物Agent] 并行执行
  每完成一个资产 → 用户确认/修改 → 下一个

Phase 2: 结构规划（1分钟）
  确认的资产 → 卷纲 → 章纲
  用户可逐卷确认

Phase 3: 正文创作（持续）
  章纲 → 写作Agent → 校验Agent → 用户确认 → 下一章
  每10章自动回顾一致性
```

## 第五部分：知识库设计分析

### 当前实现

- `KnowledgeAsset` + `RagChunk` 双表结构
- sqlite-vec 做向量存储
- BM25 + 向量混合检索（RRF融合）
- **问题**：MiMo不支持embedding API，RAG实际降级为纯BM25

### 推荐知识库架构

```
知识库
├── 结构化知识（数据库表）
│   ├── 世界观库 → Worldview表
│   ├── 人物库 → Character表
│   ├── 势力库 → 新增Faction表
│   ├── 功法库 → 新增PowerSystem表
│   └── 时间线库 → 新增Timeline表
├── 非结构化知识（RAG）
│   ├── 章节内容 → RagChunk
│   ├── 资料文档 → KnowledgeAsset
│   └── 用户笔记 → Memory
└── 元知识（关系图）
    ├── 人物关系 → CharacterRelation
    ├── 势力关系 → 新增FactionRelation
    └── 事件因果 → 新增EventChain
```

### Embedding策略

| 方案 | 适用场景 | 成本 |
|------|---------|------|
| 本地embedding（Ollama） | 开发/离线 | 免费但质量一般 |
| DeepSeek embedding | 生产环境 | 低成本高质量 |
| MiMo embedding | 如果MiMo支持 | 最优（已有API Key） |

### Rerank策略

当前没有rerank。推荐：
1. **第一阶段**：BM25 + 向量检索，取top 20
2. **第二阶段**：用LLM对20个候选做相关性打分，取top 8
3. **第三阶段**：按类型多样性重排（确保不全是同一类知识）

## 第六部分：记忆系统分析

### 当前实现

四层记忆模型设计优秀：
- `PERMANENT`（≥8）：世界规则、核心设定
- `LONG_TERM`（≥6）：当前卷主线、人物关系
- `SHORT_TERM`（≥4）：近3章事件
- `TEMPORARY`（<4）：当前场景

自动衰减 + 合并 + 清理机制完善。

### 问题

| 问题 | 说明 |
|------|------|
| 记忆写入质量 | 依赖LLM提取，质量不稳定 |
| 记忆检索不够智能 | 只按importance排序，缺少语义相关性 |
| 角色记忆未独立 | 所有记忆存在一张表，没有按角色维度隔离 |
| 用户无法编辑记忆 | 用户不能手动修正AI提取的错误记忆 |

### Memory Layer 架构优化

```
Memory System
├── Event Memory（事件记忆）
│   ├── 章节事件摘要（自动生成）
│   ├── 关键转折点（标记）
│   └── 伏笔/钩子状态（跟踪）
├── Entity Memory（实体记忆）
│   ├── 角色状态快照（每章更新）
│   ├── 势力关系变化（跟踪）
│   └── 世界观规则变更（记录）
├── Style Memory（风格记忆）
│   ├── 作者偏好的表达方式（学习）
│   ├── 角色语言特征（累积）
│   └── 场景描写模式（归纳）
└── Meta Memory（元记忆）
    ├── 用户修改记录（学习用户偏好）
    ├── 质量反馈（哪些段落被修改过）
    └── 创作意图（用户的目标和偏好）
```

## 第八部分：交互体验分析

### 用户旅程图（完成一本小说创作）

```
步骤1: 书架 → 点击"创建"                    [1次点击]
步骤2: 选择模式 → 填写标题/灵感              [2-3次点击]
步骤3: 等待Pipeline生成全部资产              [等待5-10分钟, 0次交互]
步骤4: 进入工作台 → 看到14个tab              [1次跳转, 认知冲击]
步骤5: 逐个tab查看生成结果                   [7-10次切换]
步骤6: 发现世界观不够详细 → 手动编辑         [3-5次点击]
步骤7: 发现人物卡信息不足 → 手动补充         [5-8次点击]
步骤8: 回到写作tab → 选择章节 → 开始写作     [3次点击]
步骤9: 续写 → 等待AI → 确认/修改            [1次等待 + 1次确认]
步骤10: 每章重复步骤9                        [N章 × 2次操作]
步骤11: 导出                                [2次点击]
```

**总交互成本**：
- 初始化阶段：~15次点击 + 5-10分钟等待
- 每章写作：2次操作（续写 + 确认）
- 一本100章小说：~215次操作

### 痛点

| 痛点 | 影响 |
|------|------|
| Pipeline生成期间用户无法操作 | 浪费5-10分钟 |
| 14个tab让用户不知道先看什么 | 学习成本高 |
| 世界观/人物编辑后没有"回到写作"的快捷路径 | 导航断裂 |
| 续写等待时没有进度提示 | 用户焦虑 |
| 角色卡展开/收起不直观（刚修复） | 信息获取困难 |

## 架构评审（Architecture Review）

### DDD设计
- **优点**：Novel作为聚合根，20+子实体通过级联删除保证一致性
- **问题**：缺少领域事件，实体间通信通过直接调用而非事件驱动

### 模块划分
- **优点**：Routes → Services → Prisma 三层清晰
- **问题**：`PipelineService`承担过多职责（21个子模块），应拆分为独立的Phase服务

### Agent设计
- **现状**：无Agent架构，硬编码Pipeline
- **建议**：将PhaseContext标准化为Tool接口，为Agent化预留

### 工作流设计
- **优点**：5阶段Pipeline设计合理，支持暂停/恢复
- **问题**：缺少并行执行能力，assets阶段的3个生成任务本可并行

### Prompt工程
- **优点**：writingRules.ts的16条铁律设计专业
- **问题**：prompt分散在多个文件中，缺少统一管理和版本控制

### RAG架构
- **优点**：BM25+向量混合检索+RRF融合是业界最佳实践
- **问题**：当前embedding服务不可用（MiMo不支持），降级为纯BM25

### Knowledge Base
- **优点**：KnowledgeAsset+RagChunk双表结构灵活
- **问题**：缺少结构化知识（势力、功法、法宝等独立实体）

### Memory System
- **优点**：四层衰减模型设计优秀，是真正的技术壁垒
- **问题**：记忆写入质量依赖LLM，缺少人工校正机制

### 前端架构
- **优点**：CSS Token系统设计良好，3列写作布局合理
- **问题**：NovelWorkspace 1100行+15个useState，需要拆分；混合内联/CSS样式

### 数据库设计
- **优点**：29个模型覆盖全面，索引策略合理
- **问题**：JSON字段过多（20+个），绕过数据库级校验；CharacterRelation用name而非ID引用

## 五重身份总结

**起点中文网编辑视角**：
> 这个工具能帮作者解决"日更焦虑"，但14个tab的界面会让90%的作者在第一次使用时就放弃。建议参考橙瓜码字的极简设计。

**番茄小说编辑视角**：
> 拆书模仿是杀手级功能——番茄作者最需要的就是"像XXX那样写"。但需要加强类型适配（都市/玄幻/悬疑的写法完全不同）。

**AI产品总监视角**：
> 技术架构过度设计。29个数据库模型、45个服务文件、21个pipeline模块——这是一年后的产品，不是MVP。建议砍掉50%的功能，聚焦"灵感→大纲→写作"核心链路。

**AI架构师视角**：
> Pipeline架构是当前的最佳选择，但需要为Agent化预留接口。建议将PhaseContext标准化为Tool接口，为未来多Agent协作做准备。四层记忆系统设计优秀，是真正的技术壁垒。

**小说作者视角**：
> 我只想快速把灵感变成大纲，然后一章一章写下去。现在的系统让我在"世界观不够详细"和"人物卡信息不足"之间来回跳转，写作的flow被打断了。我需要的是：输入灵感 → 立刻看到第一章 → 边写边完善设定。
