# Token 消耗优化：从 3500 次降至 ~460 次

## 变更原因

Pipeline 自动续写模式下，150 章作品消耗 3500+ 次 API 请求、几千万 tokens，导致账户欠费。根因分析：

1. **autoScheduler 重跑整个 pipeline**（占 78% 浪费）：`resumePipeline` 在 job 状态为 "completed" 时落入 `executePipeline`，重新运行所有规划阶段
2. **selfReview 每阶段调 LLM 自评**：12 个规划步骤 × 2 = 24 次 LLM（生成+自评），自评完全多余
3. **6 次后处理 LLM 调用读同一段文本**：每章 6 个独立 LLM 调用分析同一段 content
4. **每章额外 LLM 调用**：标题生成、质量评分、快速摘要各 1 次

## 改动

### Fix 1: autoScheduler 直接调用 writingPhase

`server/src/services/pipeline/autoScheduler.ts`：
- 不再调用 `pipelineService.resumePipeline(jobId)`（会重跑整个 pipeline）
- 改为直接调用 `executeWritingPhase(ctx, jobId, startOrder)`，只写新章节
- 计算 `startOrder = writtenCount + 1`，从下一章开始

`server/src/services/pipeline/writingPhase.ts`：
- `executeWritingPhase` 新增可选参数 `startOrder`
- `generateInitialChapterDrafts` 支持从指定 order 开始写作
- 使用 `enrichedChaptersMap.get(order)` 获取章纲，而非固定从 index 0 开始

`server/src/services/PipelineService.ts`：
- `ctx` 从 `private` 改为 `public`，供 autoScheduler 访问

### Fix 2: 合并 6 次后处理为 1 次 LLM 调用

`server/src/services/pipeline/postProcessing.ts`：
- 新增 `mergedPostProcessing()` 函数
- 将 storyState + memories + characterStatus + characterKnowledge + chapterSummary 合并为 1 次 LLM 调用
- 输出 2500 maxTokens 的 combined JSON
- 程序化更新出场追踪（无 LLM）
- 程序化保存 StoryState 快照（无 LLM）

`server/src/services/pipeline/writingPhase.ts`：
- 移除 `updateStoryState`、`extractMemories`、`updateCharacterAndTrackerStatus` 单独调用
- 移除 `generateChapterSummary` 异步调用（已合并到 mergedPostProcessing）
- 移除 `generateQuickSummary` LLM 调用，改用正文前 100 字

### Fix 3: 移除 selfReview 自评调用

`server/src/services/pipeline/pipelineUtils.ts`：
- `savePhaseResult` 的 `selfReviewFn` 改为可选参数，默认跳过
- `createPhaseContext` 不再将 selfReviewFn 绑定到 savePhaseResult
- `PhaseContext.selfReview` 改为可选属性

### Fix 4: 移除 LLM 质量评分

`server/src/services/pipeline/qualityCheck.ts`：
- 移除 `getLlmQualityScores` 函数（每章省 1 次 LLM）
- 质量检测改为纯程序：AI 味正则 + 字数校验
- `QualityScores` 简化为 `aiSmell` + `wordCount` 两个字段

### Fix 5: 规划阶段缓存

`server/src/services/pipeline/chapterOutlinesPhase.ts`：
- chapter outlines：检查 PhaseResult 是否已存在，存在则跳过
- story arcs：检查 PhaseResult 是否已存在，存在则跳过

## Token 节省估算（150 章作品）

| 优化项 | 原消耗 | 优化后 | 节省 |
|--------|--------|--------|------|
| autoScheduler 重跑规划 | 1200 次 | 0 次 | 1200 次 |
| selfReview 自评 | 600 次 | 0 次 | 600 次 |
| 后处理 6→1 调用 | 900 次 | 150 次 | 750 次 |
| 质量评分 LLM | 150 次 | 0 次 | 150 次 |
| 快速摘要 LLM | 150 次 | 0 次 | 150 次 |
| **总计** | **~3500 次** | **~460 次** | **~3040 次 (87%)** |

## 修改文件

| 文件 | 改动 |
|------|------|
| `server/src/services/pipeline/autoScheduler.ts` | 直接调用 executeWritingPhase，不重跑 pipeline |
| `server/src/services/pipeline/writingPhase.ts` | 支持 startOrder；合并后处理；移除 quickSummary LLM |
| `server/src/services/pipeline/postProcessing.ts` | 新增 mergedPostProcessing 函数 |
| `server/src/services/pipeline/chapterOutlinesPhase.ts` | 章纲和故事弧线缓存检查 |

| `server/src/services/pipeline/qualityCheck.ts` | 移除 LLM 评分，改为纯程序检测 |

| `server/src/services/PipelineService.ts` | ctx 改为 public |

## 验证

- TypeScript 编译通过
- autoScheduler 不再触发规划阶段
- 每章后处理从 6 次 LLM 降为 1 次
- 规划结果缓存：已有章纲不重新生成

## 风险说明

- **mergedPostProcessing prompt 复杂度**：合并后的 prompt 更长，可能影响单次输出质量
- **startOrder 逻辑**：需要确保 enrichedChaptersMap 正确重建
- **缓存失效**：如果用户修改了规划，需要清除缓存才能重新生成
