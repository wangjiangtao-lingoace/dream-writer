# Token 优化：规划驱动、按需组装

## 变更原因

Pipeline 写作阶段每章消耗 ~15,000 input tokens（全量 JSON 传递），150 章小说累计 ~250 万 input tokens，其中 90% 是与本章无关的冗余数据。本次优化将每章写作输入降至 ~1,500 tokens，降幅 90%。

## 三阶段改动

### 阶段一：精简上下文传递

- 重构 `generateChapterDraft()` 的 prompt 组装逻辑，用精简上下文替代全量 JSON
- `previousChapters` 类型从 `{content}` 改为 `{summary, ending}`，前文回顾从截断全文改为 100 字概要 + 章末 300 字
- 新增 `generateQuickSummary()` 轻量概要生成（maxTokens=150）

### 阶段二：章级概要自动生成

- 新增 `ChapterSummary` Prisma 模型，存储结构化章级概要（summary, keyEvents, characterStates, endingState, newHooks, resolvedHooks）
- 每章写完后自动生成结构化概要存入 DB（异步不阻塞）
- `ContextAssembler` 优先从 ChapterSummary 表加载前章概要，回退到 Chapter 表

### 阶段三：ContextAssembler 架构重构

- 新建 `contextAssembler.ts`，封装 `ContextAssembler` 类
- 按需加载：只加载本章出场角色精简卡（非全部角色）、世界观摘要（200 字截断）、风格精简约束（10 行）、前 N 章概要
- `writingPhase.ts` 使用 `assembler.assembleForChapter()` 替代手动数据加载，删除 5 个冗余函数

## 修改文件

| 文件 | 改动 |
|------|------|

| `server/src/services/pipeline/contextAssembler.ts` | **新建** — ContextAssembler 类 + buildCompactContext 导出函数 |

## Token 消耗对比

| 指标 | 优化前 | 优化后 | 降幅 |
|------|--------|--------|------|
| 每章写作 input tokens | ~15,000 | ~1,500 | 90% |
| 150 章累计 input tokens | ~250 万 | ~30 万 | 88% |
| 上下文组成 | 全量 JSON（世界观+人物+风格+大纲） | 章纲 + 本章角色精简卡 + 世界观摘要 + 风格约束 10 行 + 前章概要 |

## 风险说明

- **章纲质量依赖**：精简上下文后，章纲（enrichedChapter）成为唯一详细蓝图，章纲质量直接影响生成质量
- **角色信息精简**：只传 name/role/identity/motivation/arcSummary，可能丢失部分人物细节
- **世界观截断**：summary 和 rules 各限 200 字，复杂世界观可能丢失次要设定
- **回退方案**：如果生成质量下降，可将 ContextAssembler 的截断限制调大，或在 buildCompactContext 中增加更多上下文段
